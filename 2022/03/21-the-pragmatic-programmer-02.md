# 실용주의 프로그래머 📚 2장.실용주의 접근법

## 3줄 요약

- **ETC, DRY, 직교성**은 코드리뷰 요청 전에 자체 코드리뷰로 질답해 볼 것. 

```
- 이 코드는 바꾸기 쉬운 코드인가요?
- 이 코드를 수정할 때 주석 혹은 다른 중복 코드를 수정해야 하나요? (지식이 중복되는 코드가 있는지 확인해 봅니다.)
- 특정 기능에 대한 요구사항을 대폭 수정할 경우, 몇 개의 모듈이 영향을 받는가? (1개가 아니라면 좀 더 고민해 봅니다.)
```

- 추정에서 사용하는 단위가 결과의 해석에 차이를 가져온다.
- 예광탄과 프로토타입의 차이를 알고 상황에 따라 다르게 적용해 본다.

## 책에서 기억하고 싶은 내용을 써보세요.

1. **좋은 설계의 핵심**은 ETC(바꾸기 쉽게) 만드는 것이다.

  - 결합도 줄이기
  - 단일 책임 원칙
  - 이름 짓기
2. **DRY: 중복하지 말라** 
  - 코드의 어떤 측면 하나를 바꿔야 할 때 바꿔야 하는 것이 `문서(주석), DB 스키마, 스키마 구조 등`이라면 그 코드는 DRY 위반이다. 
  - 코드의 중복이 대표적이지만, `모든 코드의 중복이 지식의 중복`은 아니다. 나이를 계산하는 함수와 양을 계산하는 함수가 있다면 코드가 같아도 지식이 다르기 때문이다.
  - 표현상의 중복은 API 내부에서 mock 데이터 지원하는 툴을 쓰면 된다. API 외부에서는 OpenAPI와 같은 문서화를 통해 연동할 수 있다. 
3. **직교성(독립성, 결합도 줄이기)** 을 가진 시스템을 만들라.
  - 관련 없는 것들 간에 서로 영향이 없도록 하라. 
  - 단일하고 잘 정의된 목적을 가진 독립적인 컴포넌트다.
  - 스스로 묻기: '특정 기능에 대한 요구사항을 대폭 수정할 경우, 몇 개의 모듈이 영향을 받는가?' -> 1개여야 함.
  - 장점: 생산성 향상, 리스크 감소
4. 코딩
- 코드의 결합도를 줄여라. (객체의 상태를 바꿔야 한다면 객체가 직접 상태를 바꾸게 하라)
- 전역 데이터를 피하라.
- 유사한 함수를 피하라. (시작, 끝은 같지만 중간이 다르다면 구조에 문제가 있다는 징후이다.)
5. 객체 지향 언어 VS 함수형 언어
- **객체 지향 언어**
  - 불투명한 결합을 만들 가능성을 높이는 기능(다중 상속, 예외, 연산자 오버로딩, 상속 오버라이딩)이 많다. 
  - 클래스가 관심사에 집중하지 않으면 인터페이스가 엉망으로 망가질 수 있다. 
- **함수형 언어**
  - 작고 독립적인 함수를 많이 마니들어 조합하여 문제를 해결하도록 유도한다.
  - 함수는 일반적으로 데이터를 반환하는데, 한 함수의 결과가 다른 함수의 입력이 되면 결합이 발생한다.
  - 주의하지 않으면 데이터 포맷을 바꾸는 바람에 이어지는 변환을 망가트릴 수 있다.
6. 예광탄 VS 프로토타이핑
- **프로토타입**
  - 최종 시스템의 어떤 특정한 측면을 탐사해 보는 것.
  - 실험 과정이므로 모두 버리고 교훈삼아 새로 작성한다. (현실에 프로토타입이 반영되지 않을 수 있음)
  - 실제 제품보다 훨씬 적은 비용으로 만들 수 있다.
- **예광탄**
  - 대강 구현한 알고리즘과 단순하지만 동작하는 UI로 구성한다.
  - 자리만 만들어 두었던 루틴을 채우면서 새 기능을 차근차근 추가한다. 
  - 첫 번째 예광탄 코드가 앞으로도 그 방식 그대로 동작한다.
  - 세부 사항을 포기할 수 없다면 예광탄 방식이 더 적절하다.

- 요구사항 확인 > 위험도 높은 부분 우선 하기 > 설계, 구현, 통합 > 검증

- **ETC, DRY, 직교성**은 코드리뷰 요청 전에 자체 코드리뷰로 질답해 볼 것. 
- `모든 코드의 중복이 지식의 중복`은 아니다.: 이 부분이 신선한 충격이었다. 중복만 보면 무조건 리뷰를 달았었는데, 코드 전체를 보고 판단해야겠다는 생각을 하게 되었다.
